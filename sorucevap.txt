# KINDERLab ROBOTICS PROJESI - GUNCEL SORU-CEVAP REHBERI

Bu dosya, proje hakkinda teknik mulakat, devir ve sunum sorularina hizli ve dogru cevap vermek icin guncel tutulur.

---

## 1) PROJE GENEL BAKIS

### S1: Bu proje nedir?
**C:** KinderLab Robotics icin gelistirilmis full-stack bir web platformudur. Ana alanlar:
- E-ticaret (urun, sepet, siparis)
- Icerik yonetimi (blog, basin bultenleri, medya kapsami, etkinlikler)
- Egitim odakli sayfalar ve kurs kayitlari
- Admin paneli ile operasyon yonetimi

### S2: Projenin hedef kitlesi kim?
**C:** Ogretmenler, veliler, egitim kurumlari ve kurum ici admin ekibi.

### S3: Projenin su anki durumu ne?
**C:** Guvenlik ve veri butunlugu tarafi guclendirildi. Payment gateway (iyzico/stripe/paytr vb.) henuz bagli degil, ama payment domain altyapisi hazir. Admin dashboardda orders/users/contacts/blog/press/media/events bolumleri parcali bilesenlere ayrildi.

### S4: Neden bu kadar moduler yapi tercih edildi?
**C:** Monolit route dosyasi yerine moduler route/middleware/service yapisi ile degisiklik etkisi azaltildi ve test/yayina alma riski dusuruldu.

---

## 2) TEKNIK MIMARI

### S5: Teknik mimari nedir?
**C:**
```text
Frontend (React + Vite + TypeScript)
        <->
Backend (Node.js + Express)
        <->
Database (SQLite)
```

### S6: Frontend teknolojileri neler?
**C:** React 18, TypeScript, Vite, React Router, Tailwind CSS, TanStack Query, React Hook Form, Zod, TinyMCE, Radix UI.

### S7: Backend teknolojileri neler?
**C:** Node.js, Express, sqlite3, multer, sanitize-html, uuid.

### S8: Veritabani neden SQLite?
**C:** Bu asamada tek sunucu + orta trafik senaryosu icin operasyonel olarak basit. Ayrica test ortamlari icin dosya tabanli DB kurulumu hizli.

### S9: Ana dizin yapisi nasil?
**C:**
```text
src/      -> frontend sayfa, context, servisler
server/   -> express app, route, middleware, service, db
public/   -> statik varliklar
```

## 3) BACKEND VE API

### S10: Auth modeli nasil?
**C:** Bearer token tabanli. Token dogrulandiktan sonra middleware kullaniciyi DB'den yeniden dogrular (token tek basina yeterli degil).

### S11: Public ve protected endpoint ayrimi nasil?
**C:**
- Public: login/register, contact post, public CMS get endpointleri
- Protected: kalan endpointler (kullanici/admin), role guard middleware ile kontrol edilir

### S12: CMS endpointleri nasil ayrildi?
**C:**
- Public CMS list/detail endpointleri sadece `published` icerik dondurur
- Admin list endpointleri ayridir:
  - `/api/admin/blog`
  - `/api/admin/press-releases`
  - `/api/admin/media-coverage`
  - `/api/admin/events`

### S13: E-ticaret endpointleri neler?
**C:**
- `/api/orders`
- `/api/orders/my`
- `/api/orders/:id/status`
- `/api/orders/:id/payment-status`
- `/webhooks/carrier/orders/:id/status`

### S14: Payment-status endpointleri ne ise yariyor?
**C:**
- `GET /api/orders/:id/payment-status`: owner/admin payment durumu + deneme loglarini gorur
- `PUT /api/orders/:id/payment-status`: admin payment durumunu gunceller

### S15: Upload endpointleri nasil?
**C:** Blog/press/media/events icin hem kalici hem temp image upload endpointleri var. Multer file type ve boyut validasyonu yapar.

---

## 4) E-TICARET VE PAYMENT DOMAIN

### S16: Siparis olusturma akisinda guvenlik ne durumda?
**C:** Siparis kalemleri backend tarafinda dogrulaniyor. Urun kimligi backend katalog servisinden (`server/services/productCatalogService.js`) kontrol ediliyor.

### S17: Client tarafi fiyat manipule edilirse ne oluyor?
**C:** Reddediliyor. Total ve item butunlugu backend hesaplamasi ile dogrulaniyor.

### S18: Fulfillment akisi payment ile nasil bagli?
**C:** Siparis fulfillment gecislerinde `payment_status = paid` zorunlu. Admin sadece `received/preparing/shipping` guncelleyebiliyor; `shipping` icin carrier + tracking zorunlu. `delivered` varsayilan olarak carrier webhook ile geliyor.

### S19: Payment gateway yokken hangi altyapi hazir?
**C:**
- Orders tablosunda payment kolonlari
- `payment_attempts` tablosu (deneme kayitlari)
- `payment_events` tablosu (event/webhook log ve idempotency)
- Provider-agnostic payment service iskeleti (`server/services/paymentService.js`)

### S20: Gercek PSP baglanmadiysa ne yapabiliyoruz?
**C:** Manual/admin payment status update ile operasyon akisi test edilebiliyor; fulfillment guard gercekten calisiyor. Admin tarafinda payment attempts goruntuleme ve payment status guncelleme yapilabiliyor.

### S21: Neden provider-agnostic payment service kuruldu?
**C:** PSP secimi netlesmeden once domain modeli sabitlensin diye. Boylece iyzico/stripe/paytr secimi sonrada yapildiginda kodu bastan yazmak gerekmiyor.

### S23: Migrasyon mantigi nasil calisiyor?
**C:** `server/database/database.js` icindeki `runMigrations()` adimlari startup'ta calisir. Kolon/table yoksa ekler; mevcut veriyi bozmadan ilerler.

### S24: Kritik yazma akislarinda transaction var mi?
**C:** Evet. `runInTransaction` ile siparis gibi coklu yazma adimlari atomik calisir.

### S25: DB dosyasi repoda var mi?
**C:** Evet, `server/database/kinderlab.db` bilincli olarak tracked. Secret/gercek production veri konmamalidir.

## 6) GUVENLIK

### S26: Temel guvenlik katmanlari neler?
**C:**
- Token auth + DB re-validation
- Role-based admin guard
- Public/protected endpoint ayrimi
- Input sanitize (backend: sanitize-html / frontend: DOMPurify)
- Upload type/size/path guvenligi

### S27: Rate-limit var mi?
**C:** Evet.
- Login rate-limit
- Public contact endpoint rate-limit
- Memory bloat riskine karsi bounded/prune mantigi eklendi

### S28: `TRUST_PROXY` neden onemli?
**C:** `x-forwarded-for` sadece reverse proxy arkasinda guvenilir. Bu nedenle `TRUST_PROXY=true` sadece dogru topolojide acilmali.

### S29: Demo endpointleri nasil kontrol ediliyor?
**C:** `ENABLE_DEMO_ENDPOINTS=false` ise demo/reset endpointleri kapali; production icin zorunlu kapali tutulmali.

## 9) PSP SECILMEDEN ONCE VE SONRA

### S38: PSP secilmeden once yapilanlar yeterli mi?
**C:** Evet, domain hazirligi tamamlandi. Kalan is PSP adapter + webhook imza dogrulama + checkout redirect akisi.

### S39: PSP secildiginde ilk teknik adim ne olmali?
**C:** Sandbox entegrasyonu: `create-session` + `webhook` + idempotent event isleme, sonra success/failure/cancel senaryolari.

### S40: Canliya geciste en kritik risk ne?
**C:** Webhook dogrulama ve idempotency hatalari. Bu nedenle event kaydi ve tekrar event isleme guvenceleri zorunlu.

## 10) GUNCEL TEKNIK BORC OZETI

### S41: Su an acik kalan ana teknik borclar neler?
**C:**

- Upload/content unhappy-path testlerinin daha da artirilmasi
- Account UI integration testlerinin artirilmasi

### S42: Bir sonraki en mantikli adim ne?
**C:** PSP secimi netlesince sandbox adapteri baglamak ve webhook tabanli odeme kapanisini tamamlamak.
